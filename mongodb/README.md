## Теоретическая часть

MongoDB — документоориентированная NoSQL СУБД, хранящая данные в формате BSON (бинарный JSON).  
Основные понятия:
- **База данных** — контейнер коллекций.
- **Коллекция** — группа документов (аналог таблицы в SQL), но без фиксированной схемы.
- **Документ** — набор пар «ключ-значение», может содержать вложенные объекты и массивы.
- **Агрегационный пайплайн** — последовательность этапов обработки документов (аналог `SELECT ... JOIN ... GROUP BY` в SQL). Важнейшие этапы:
  - `$match` — фильтрация;
  - `$unwind` — развёртывание массива в отдельные документы;
  - `$lookup` — левое внешнее соединение с другой коллекцией;
  - `$group` — группировка с агрегатными функциями (`$sum`, `$avg`);
  - `$project` — формирование выходных полей;
  - `$sort`, `$limit` — сортировка и ограничение количества.

В работе использовалась облачная платформа MongoDB Atlas (бесплатный кластер M0) либо локальная установка.

### Ключевые фрагменты кода

#### 1. Вставка заказа с вложенным массивом `items`
```javascript
db.orders.insertOne({
    _id: 1,
    user_id: 1,
    order_date: new Date("2026-04-15"),
    status: "completed",
    items: [
        { product_id: 1, quantity: 1, price: 75000 },
        { product_id: 2, quantity: 2, price: 1500 }
    ]
});
```

#### 2. Агрегационный пайплайн — отчёт по категориям товаров
```javascript
db.orders.aggregate([
    { $unwind: "$items" },
    { $lookup: {
        from: "products",
        localField: "items.product_id",
        foreignField: "_id",
        as: "product_info"
    }},
    { $unwind: "$product_info" },
    { $group: {
        _id: "$product_info.category",
        total_quantity: { $sum: "$items.quantity" },
        total_revenue: { $sum: { $multiply: ["$items.quantity", "$items.price"] } },
        avg_price: { $avg: "$items.price" }
    }},
    { $sort: { total_revenue: -1 } },
    { $project: {
        _id: 0,
        category: "$_id",
        total_quantity: 1,
        total_revenue: 1,
        avg_price: { $round: ["$avg_price", 2] }
    }}
])
```

#### 3. Топ-3 пользователей по сумме заказов (аналог SQL)
```javascript
db.orders.aggregate([
    { $unwind: "$items" },
    { $group: {
        _id: "$user_id",
        total_spent: { $sum: { $multiply: ["$items.quantity", "$items.price"] } }
    }},
    { $sort: { total_spent: -1 } },
    { $limit: 3 },
    { $lookup: {
        from: "users",
        localField: "_id",
        foreignField: "_id",
        as: "user"
    }},
    { $unwind: "$user" },
    { $project: { _id: 0, full_name: "$user.full_name", total_spent: 1 } }
])
```

## Выводы
1. Документная модель MongoDB позволяет эффективно хранить связанные сущности (заказ и его товары) внутри одного документа, избавляя от необходимости множественных соединений при чтении.
2. Агрегационные пайплайны в MongoDB, несмотря на более громоздкий синтаксис по сравнению с SQL, обладают высокой гибкостью и позволяют выполнять сложные аналитические запросы.

## Ответы на контрольные вопросы
1. **Как в MongoDB представлена связь «заказ-товары» в отличие от PostgreSQL?**  
   В PostgreSQL используется отдельная таблица `order_items`, соединяемая через внешние ключи. В MongoDB товары хранятся в виде вложенного массива `items` внутри документа заказа, что позволяет получить все данные заказа одним чтением без `JOIN`.

2. **В каком случае документная модель удобнее реляционной? Приведите пример из работы.**  
   Документная модель удобнее, когда объект содержит в себе иерархически связанные части. В нашем примере веб-страница с подробностями заказа может быть построена из одного документа MongoDB, тогда как в SQL потребовалось бы соединение трёх таблиц.

3. **Какие операции в MongoDB оказались сложнее/проще по сравнению с SQL?**  
   Проще: вставка сложных вложенных документов (одна операция вместо нескольких `INSERT`). Сложнее: агрегация с `$unwind` и `$lookup` требует большего количества этапов и понимания пайплайна, в то время как SQL-запрос с `JOIN` и `GROUP BY` выглядит компактнее и привычнее.

4. **Что такое `$unwind` и зачем он нужен?**  
   `$unwind` – оператор агрегации, который разворачивает массив внутри документа, создавая отдельный документ для каждого элемента массива. Это необходимо для того, чтобы применять фильтрацию, группировку или соединение к отдельным элементам массива (например, подсчитать общую выручку по всем товарам).
